组件选型与依据

每个组件均从横向扩展性与灾难恢复 / 高可用性两个维度进行选型论证。

AWS 组件 选型理由 扩展性贡献 高可用贡献
自定义 VPC
网络隔离
 独立网络边界，完整控制子网、路由与安全规则 大 CIDR 地址空间支持多子网横向扩展 双 AZ 跨可用区是故障容错的网络基础
Application Load Balancer
弹性
 L7 请求分发，内置健康检查，AWS 托管无单点 自动将流量分发到 ASG 运行的所有实例 持续健康检查，故障实例自动摘除
Auto Scaling Group
弹性
 按负载自动调整容量，min 2 / max 8 实例 CPU>60% 自动扩容，CPU<30% 缩容，峰值最多 8 实例 min=2 保证双 AZ 各一实例，故障实例自动替换
Elastic Beanstalk
编排
 托管 ALB + ASG + EC2 编排，版本化部署 自动配置和再平衡扩展栈，无需人工介入 版本化 Bundle（v5）支持快速回滚
RDS MySQL Multi-AZ
高可用
 托管关系型数据库，自动备份与补丁 实例纵向扩展 + 可选读副本 第二 AZ 同步热备，自动故障转移，时间点恢复
自定义 AMI
弹性
 预装 Python / Flask / gunicorn，一致性启动 新实例秒级就绪，扩容无需等待 bootstrap 故障实例从 AMI 确定性重建，快速恢复
Security Groups
安全
 最小权限层级访问控制 规则自动应用于 ASG 启动的每个实例 RDS 仅接受来自 EC2 安全组的 3306，互联网不可达

技术难点与解决方案

  • RDS 安全组设计：仅允许 EC2 安全组 ID 访问 3306，数据库对互联网完全不可达
  • AMI 预烘焙策略：预装 Flask + gunicorn + PyMySQL，新实例无需 bootstrap 延迟即可立即服务
  • 多 AZ 子网绑定：ALB 与 ASG 同时跨 us-east-1a / 1b，单 AZ 故障不影响服务
  • CloudWatch 双阈值策略：60% 扩容 / 30% 缩容，避免频繁伸缩（抖动）
  • Lab 环境约束处理：使用 LabRole / LabInstanceProfile，EC2 部署于公有子网（生产环境应置于私有子网 + NAT Gateway）

架构局限性与生产优化建议

  • 将 EC2 迁移至私有子网，ALB 保留公有子网，增加 NAT Gateway
  • ALB 添加 AWS ACM 证书，强制 HTTP → HTTPS 重定向
  • Port 22 SSH 访问限制至 Bastion Host 或管理员 IP 段
  • 生产区域切换至 ap-southeast-2（Sydney），降低中国澳大利亚用户延迟
  • 增加请求数 / 延迟 Target Tracking 策略，补充 CPU 指标的不足
