1 · 问题背景：为什么信息越多越难找

2025年5月，我从中国到悉尼开始软件系统工程硕士学习。此后 10 个月里，我同时处理以下认知负荷：一个贯穿全年的毕业设计项目（UGMAS 煤矿监测系统，基于两篇学术论文的 TCA/FSV 框架）、五门课程作业（.NET 应用开发、云计算 AWS、企业架构 Salesforce、UNIX 系统编程、风险管理）、以及求职准备。

多线并行本身不是问题。真正的问题是：信息存在四个互不相通的容器里——课程笔记在 Craft，临时想法在 Apple Notes，项目文档在 Notion，技术摘抄散落在桌面。每次开始新任务，第一件事是花 15–20 分钟回忆「那个内容记在哪里」。

核心矛盾

信息管理领域有一个著名悖论：笔记越多，找到所需内容反而越难。这是因为传统的文件夹-文档组织模式是「存储优先」而非「检索优先」的——你在记录时按主题分类，但检索时需要跨主题联想。两者的认知模式根本不同。Forte（2022）将此称为「数字囤积」问题：大量保存但从不复用。

PKM（Personal Knowledge Management）领域对此有系统性研究。Zettelkasten 方法（Ahrens, 2017）提出以「原子化笔记 + 双向链接」代替层级文件夹，使笔记网络能够反映真实的思维联系而非人为的分类边界。Obsidian 是目前对这套理念落地最完整的工具，其 Graph View 功能使知识网络的拓扑结构可视化，为知识管理提供了结构化的反思界面。

2 · 系统架构：Obsidian 作为知识图谱基础设施

2.1 设计原则

Vault「Everest」的设计遵循三个原则：

领域优先而非类型优先。文件夹按生活领域（UTS / 工作 / 个人 / 财务）而非信息类型（笔记 / 资料 / 任务）组织。理由是：检索时脑子里浮现的是「那个 AWS 的内容」而不是「那是笔记还是资料」。

MOC 而非完全扁平。每个领域顶层设一个 ！Index.md（Map of Content），用双向链接维护该领域的知识地图。MOC 不需要手动维护所有条目，可以用 Dataview 自动查询。MOC 节点因连接密度高，在 Graph View 中自然成为枢纽。

Inbox 缓冲层。所有新内容先进 📥 Inbox，定期整理归档。这消除了「该放哪里」的决策摩擦，使捕获动作接近零成本。

2.2 Vault 结构

📦 Everest/ ├── 📥 Inbox/ ← 新内容缓冲区，定期整理 ├── 🎓 UTS/ │ ├── ！Index.md ← MOC 枢纽，所有课程入口 │ ├── Semester 1/ │ │ ├── 42003 UGMAS/ ← 毕业设计，节点最密集 │ │ ├── 32998 .NET/ │ │ └── 42904 Cloud/ │ └── Semester 2/ ├── 💼 JOB/ ← 公司研究、面试题库、简历迭代 ├── 💻 Tech/ ← 技术摘录，双链密度最高 ├── 👤 Personal/ ├── 💰 Money/ ├── ✈️ Travelling/ └── 📅 Daily/ ← YYYY-MM-DD.md 日记，链接当日相关笔记

2.3 双向链接的信息论意义

普通文档的链接是单向的：A 提到 B，但 B 不知道被 A 提到。Obsidian 的双向链接使每个笔记都能看到「谁链接了我」，这在信息论上的意义是：一个笔记的被引用数近似于它在知识网络中的中心性。高中心性的笔记（如 [[TCA 算法原理]] 被 12 篇笔记链接）在 Graph View 中节点更大，视觉上立刻突出。

这与 Luhmann（1992）的 Zettelkasten 原理一致：知识的价值不在于单条笔记的质量，而在于笔记之间连接形成的语义网络密度。当网络足够大时，新笔记的归档本身就是一次「新知识与已有认知框架的冲突与协调」过程。

3 · Graph View 脑图：知识网络的可视化与分析

Graph View 是 Obsidian 的核心差异化功能，将所有笔记及其双向链接渲染为力导向图（Force-Directed Graph）。下图为 Everest Vault 在约 10 个月后的实际状态：

图1：全局知识网络 800+ 节点，多枢纽中心结构，外围散点为待链接的孤立笔记
图2：#知识 标签聚焦视图 单标签节点辐射 50+ 连线，展示跨领域概念的高聚合密度

3.1 图谱结构分析

从图1可以观察到几个典型的网络拓扑特征：

多枢纽-辐射结构（Hub-and-Spoke）。图中存在 5–6 个高连接密度的大节点（黄色），对应各领域的 MOC 索引文件。这是预期结构，说明 MOC 设计有效——它们成为跨领域导航的入口。

外围孤立节点群。图1外圈有大量点状聚集的孤立节点（蓝色小点），这是迁移过来但尚未建立双链的历史笔记。它们在图谱中可见但处于「死角」，是待处理的技术债。

#知识 标签的超级枢纽效应。图2中 #知识 标签节点辐射出约 50 条连线，远超其他标签。这说明它被过度使用——在实践中发现，过于宽泛的标签会失去分类价值，需要进一步细分为 #知识/技术、#知识/方法论 等子标签。

Graph View 的实践价值

Graph View 不只是「好看」。它的核心价值是提供了一个知识健康度的视觉指标：孤立节点过多说明迁移工作待完成；某个领域连接过密说明该领域笔记质量高；两个领域之间没有连接说明它们在你的认知中还是孤立的。这是传统文件夹系统完全无法提供的元认知信息。

3.2 使用 Dataview 自动化 MOC 维护

手动维护 MOC 链接会随笔记增多变得沉重。Dataview 插件支持用 SQL 语法自动生成动态列表，使 MOC 自我更新：

UTS/！Index.md · Dataview 自动 MOC 示例
## 本学期所有项目笔记（自动更新） ```dataview TABLE file.mtime AS "最后修改", length(file.inlinks) AS "被引用数" FROM "UTS" AND #project SORT length(file.inlinks) DESC LIMIT 10 ``` ## 本周新增笔记 ```dataview LIST FROM "UTS" WHERE file.mtime >= date(today) - dur(7 days) SORT file.mtime DESC ```

4 · Claude 作为知识处理代理：Prompt 工程实践

Obsidian 解决「存储和检索」问题，但它不能解决「原始内容 → 结构化知识」的转化问题。这是认知负荷最高的环节——面对一篇 30 页的学术论文，如何快速提炼出对自己有用的核心？如何在不失去细节的前提下生成可用的 Markdown 笔记？

这正是 Claude 嵌入工作流的位置。Claude 不是替我思考，而是加速「原始输入 → 可用格式」的转化过程，把我的精力释放到判断和连接上。

4.1 工作流四阶段

📥
捕获 — 零摩擦原始记录
碎片想法、会议内容、论文摘录先进 Apple Notes（iOS 设备随时可用，Siri 语音转文字进一步降低摩擦）。此阶段不做任何整理，只做记录。
工具：Apple Notes · Siri 语音输入
🤖
处理 — Claude 代理提炼
将原始内容粘贴给 Claude，使用标准化 Prompt 模板提炼结构、生成 Markdown 草稿、标注应建立独立笔记的术语。关键原则：由我判断筛选，Claude 加速转化，不替代决策。
工具：Claude · 定制 Prompt 模板（见 4.2 节）
🔗
归档 — Obsidian 双链组织
将 Claude 输出的笔记草稿归入对应 Vault 节点，手动补充 [[双向链接]] 连接到相关笔记。这个手动环节是刻意保留的——建立链接的过程是主动思考两个笔记关系的过程，不能外包给 AI。
工具：Obsidian · Graph View 验证连接密度
♻️
复用 — 历史知识服务新任务
新任务启动时先搜 Obsidian，再搜互联网。如果相关笔记存在，从那里出发可以节省大量重复调研时间，并保持决策与历史判断的一致性。
工具：Obsidian 全文搜索 · Dataview 查询

4.2 核心 Prompt 模板

以下是在实践中迭代稳定的三个 Prompt 模板，覆盖了工作流中最高频的三个场景：

Prompt 模板 01 · 原始笔记 → 结构化 Markdown
以下是我的原始笔记，内容比较零散： [粘贴原始内容] 请帮我完成以下工作： 1. 提炼出 3-5 个核心概念，每个用一句话解释 2. 整理成 Obsidian Markdown 格式，用 ## 分层 3. 在合适的地方用 [[概念名]] 格式标注应建立独立笔记的术语 4. 最后列出 2-3 个「待深入」的问题，写成 TODO 格式 约束： - 不添加我原始笔记里没有的观点，只做结构化整理 - 保持中文，技术术语保留英文原文 - 控制在 500 字以内
Prompt 模板 02 · 论文快速精读（用于学术文献）
我需要在 45 分钟内理解以下论文的核心贡献，以便在我的项目中使用其方法： 论文标题：[标题] 研究背景：[我的项目背景] 请按以下结构输出： 1. 核心问题（一句话） 2. 提出的方法/框架（要素+原理，不超过 200 字） 3. 关键公式或算法步骤（如有，用代码块格式） 4. 实验结果的核心数字（我需要哪些数字作为对标基准） 5. 对我项目的直接可用部分 6. 我需要注意的假设前提（哪些条件下方法会失效） 格式：直接输出，不要过渡句，每节用 ## 开头
Prompt 模板 03 · 技术选型决策辅助
我在做 [项目名称]，需要在以下方案之间做选择： 方案 A：[具体描述] 方案 B：[具体描述] 我的具体约束： - 时间：[可用时间] - 技术栈：[现有技术] - 主要评估维度：[列出 2-3 个] 请帮我分析两个方案的权衡，不需要给出「推荐」结论。 我需要理解权衡本身来做自己的决策。 输出格式：对比表格 + 每个方案的核心适用场景
关键原则

Claude 输出的内容必须经过判断再进 Obsidian，不能直接粘贴。最有效的方式是：阅读 Claude 输出后，用自己的话重写关键部分，再加上 [[双向链接]]。这个重写过程是真正的学习发生的地方。如果跳过这一步，笔记里储存的是「Claude 的理解」而非「我的理解」，在需要快速调用时会发现无法直接使用。

5 · 迁移工程：从四工具混沌到单一知识库

从 Craft + Notion + Apple Notes + 桌面文件到统一的 Obsidian Vault，这个迁移过程遇到了若干可重复的工程问题。记录于此以减少他人的重复踩坑。

Craft 导出 Markdown 后图片全部断链，![](image.png) 路径无效
不要单独移动 .md 文件。Craft 导出时 .md 与 .assets 文件夹在同目录，保持该相对路径关系直接作为 Vault 打开，图片自动恢复。若已分离，用 Consistent Attachments and Links 插件批量修复。
侧边栏被大量 .assets 文件夹污染，导航效率极低
安装 Hidden Folder 插件，配置 *.assets 模式，一键隐藏所有资源文件夹。同理可隐藏 .obsidian 等系统目录。
文件全部平铺时，侧边栏滚动出现 Electron 渲染残影和黑块
两步解决：① 设置 → 外观 → 关闭硬件加速；② 整理文件夹结构（根本解法）——平铺文件节点过多是导致渲染压力的真正原因。
编辑区两侧留白过多，笔记宽度不足，长文写作体验差
安装 Minimal 主题 + Minimal Theme Settings 插件，Editor Width 设置为 Wide 或 Max，内容铺满编辑区。
换 Vault 后已安装插件全部消失，需要重新配置
每个 Vault 的 .obsidian/ 文件夹独立存储插件和设置。快速迁移方法：将旧 Vault 的 .obsidian/ 文件夹复制到新 Vault 根目录，重启 Obsidian 即可恢复全部配置。

5.1 推荐插件清单（最小化原则）

遵循「每个插件必须解决一个明确的真实问题」原则，当前只用以下四个：

插件解决的具体问题核心配置
Hidden Folder隐藏 .assets 等污染侧边栏的系统目录添加 *.assets 规则
Minimal Theme Settings编辑区宽度和字体优化Editor Width → Wide/Max
Dataview自动生成动态文件列表，替代手动维护 MOC启用 JavaScript 查询
Consistent Attachments批量修复从 Craft/Notion 迁移的断链图片一次性修复后可卸载

6 · 项目应用案例：三个真实场景的深度记录

案例 A：UGMAS 毕业设计（最复杂案例）

UGMAS 是一个持续 12 个月、基于两篇 Q1 期刊论文的毕业设计。知识管理路径如下：用 Prompt 模板 02 精读 Wu et al.（2021）和 Wu et al.（2023），输出 TCA 算法原理.md 和 FSV 稳定性验证.md，两者在 Graph View 中成为 42003 文件夹内连接密度最高的节点。每次实现遇到问题，先搜索这两个文件，大部分问题能在里面找到线索。

三版迭代的每一次「为什么这样改」都记录在 UGMAS 迭代日志.md 中，最终报告写作时直接从笔记提取论据，报告写作时间比预估缩短约 40%。

案例 B：RAG 系统技术选型

构建多模态 RAG 系统时，面临三个关键选择：向量数据库（Chroma vs Pinecone vs Qdrant）、检索策略（纯向量 vs BM25 混合）、Rerank 方案（Cross-encoder 类型）。用 Prompt 模板 03 对每个选项做了对比分析，输出统一归入 RAG 技术选型对比.md。

这篇笔记在整个项目周期内被打开了 30+ 次，帮助在实现过程中保持决策一致性。最终选择的 Chroma + BM25 混合方案，其核心论据直接来源于这篇笔记而非重新调研。

案例 C：求职准备的知识管理

💼 JOB/ 文件夹下维护了一个「公司研究」目录，每家感兴趣的公司对应一个笔记，包含：业务方向摘要、技术栈偏好（从 JD 提取）、与个人技能的匹配分析、面试准备行动项。用 Claude 分析招聘 JD 的关键词频率和能力要求，补全了和目标岗位的技能差距图。这些笔记最终反哺了简历的三次迭代。

7 · 系统评估：有效性与局限性

7.1 有效的方面

消除信息孤岛，使跨域连接可见。最明显的收益是：不再需要记住「那个东西在哪个工具里」。Graph View 使不同课程之间的概念重叠变得可见——当 TCA 的力导向图和 D3.js 力导向图这两个概念通过 [[力导向图算法]] 节点连接时，意识到两者有共同的底层原理，这种连接在分散工具里根本不可能被发现。

减少重复调研，保持决策一致性。以 RAG 项目为例，技术选型笔记在整个项目期间被复用 30+ 次。如果没有这个系统，每次遇到选型相关问题都需要重新搜索和判断。

7.2 局限性与未解决的问题

笔记质量在时间压力下下降明显。高负荷时期写的笔记结构混乱，事后回看需要重新理解。这不是工具问题，而是「记录」和「深度思考」在时间压力下难以兼顾的现实矛盾。Claude 提炼有帮助，但无法完全弥补原始内容质量不足。

外围孤立节点是持续的技术债。迁移过来但未建立双链的历史笔记数量庞大（图1中可见），整理这些孤立节点需要大量时间，目前只能逐步消化。

Graph View 在超过 1000 节点后渲染性能下降。随着 Vault 增长，Graph View 的加载时间会增加，需要定期通过过滤器限制显示范围。

8 · 下一步：RAG 接入与知识库智能化

目前工作流中 Claude 的角色是「被动代理」——我需要主动把内容粘贴给它，它才能处理。下一个里程碑是将整个 Obsidian Vault 接入 RAG 系统，实现直接问答自己的笔记。

规划中的 RAG 接入方案

索引：将 Vault 中所有 .md 文件解析为 chunks，嵌入向量数据库（Chroma）。[[双向链接]] 作为元数据保留，在检索时用于扩展上下文——当检索到笔记 A 时，自动加载 A 链接的笔记 B 和 C 作为补充上下文，这是 Obsidian 知识库相对于普通文档集合的独特优势。

检索：BM25 + 稠密向量混合检索，与我在 RAG 项目中验证的方案相同（Hit@3 = 0.87）。

接口：本地部署，不上传原始笔记到第三方服务，保护隐私敏感内容。

这个方向代表了 PKM 工具演进的一个关键趋势：从「被动存储」到「主动智能」。Obsidian 的纯文本 + 双向链接架构，是目前最适合 RAG 接入的笔记格式——Markdown 解析成本低，双向链接提供了天然的图结构，可以用图遍历算法扩展检索上下文。
